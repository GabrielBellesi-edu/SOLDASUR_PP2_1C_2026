# Unificación de Servidores: Servidor Único en FastAPI (v4.0.2)

Este documento registra la unificación de la arquitectura de servidores para la versión **v4.0**, permitiendo que toda la aplicación (Frontend y Backend) sea servida de manera centralizada bajo un único proceso en el puerto `8000`.

---

## 1. El Problema de la Arquitectura de Servidores Múltiples

Anteriormente, para utilizar la aplicación de manera completa, se requería levantar dos servidores por separado:
1. **Frontend (Puerto 8001 / 8000):** Servido a través del módulo ligero de Python `python -m http.server`.
2. **Backend (Puerto 8000):** Servido a través de FastAPI y Uvicorn `python -m uvicorn app.main:app`.

### Desventajas de este enfoque para el cliente (SOLDASUR):
* **Complejidad de operación:** Los vendedores y usuarios de la sucursal (quienes no son programadores) debían abrir dos consolas y ejecutar dos comandos distintos, lo que aumentaba la posibilidad de cometer errores o de que una de las partes quedara inactiva.
* **Problemas de puertos:** Si el servidor estático se iniciaba accidentalmente en el puerto 8000, bloqueaba al backend de FastAPI, impidiendo que el chatbot con IA pudiera responder.
* **Problemas de CORS:** Requería configurar reglas de intercambio de recursos de origen cruzado (CORS) en el backend de Python para permitir que el frontend del puerto 8001 realizara llamadas HTTP al puerto 8000.

---

## 2. La Solución Unificada en FastAPI

Para solucionar esto, se optó por consolidar toda la entrega de la aplicación bajo el backend de FastAPI. Esto permite que el propio servidor Uvicorn actúe tanto de servidor de archivos estáticos (HTML/CSS/JS) como de servidor de API.

### Implementación Técnica:
En el archivo [main.py](../app/main.py), se habilitó el montaje de archivos estáticos en la raíz del servidor:

```python
from fastapi.staticfiles import StaticFiles

# Montar la raíz del proyecto para servir todos los archivos estáticos del frontend (HTML, JS, CSS, JSON, imágenes, etc.)
app.mount("/", StaticFiles(directory=BASE_DIR, html=True), name="frontend")
```

* **Prioridad de Rutas:** El montaje de archivos estáticos se colocó al final de la definición de todas las rutas de la API en `main.py`. Esto garantiza que los endpoints dinámicos (como `/api/chat`, `/ask`, `/start`, `/reply`) sean evaluados primero por FastAPI, evitando que el servidor confunda una llamada a la API con una solicitud de archivo estático.
* **Portabilidad Absoluta:** Al definir el directorio de recursos mediante la constante `BASE_DIR` (que apunta de forma absoluta a la raíz del repositorio), el servidor es completamente portable y se puede iniciar de forma correcta desde cualquier consola, sin importar el directorio de trabajo activo.

## 3. Resolución de Errores Críticos durante la Puesta en Marcha

Durante el despliegue del entorno y las pruebas de interacción en la versión unificada, se detectaron y corrigieron dos errores del backend de Python:

### A. Corrección de la Base de Datos SQLite de PEISA (`sqlite3.OperationalError: no such table: products`)
* **Problema:** El archivo de base de datos `embeddings/products.db` copiado de la versión anterior estaba vacío (0 bytes) y no contenía la tabla `products`. Esto provocaba que las consultas semánticas del chatbot de PEISA fallaran con un error HTTP 500 en la terminal de Uvicorn.
* **Solución:** Se creó un script en Python que lee la lista completa de productos del archivo [products_catalog.json](../data/products_catalog.json), crea la tabla `products` con el esquema necesario en SQLite y puebla los 69 productos en orden secuencial para garantizar la perfecta alineación con el índice vectorial FAISS (`products.faiss`).

### B. Prevención de Crasheos por Codificación en Windows (`UnicodeEncodeError`)
* **Problema:** El servidor de Uvicorn crasheaba al iniciar la API debido a que en el archivo [weber_expert_engine.py](../app/modules/expertSystem/weber_expert_engine.py) se imprimían en la consola de comandos emojis Unicode (como `✅` y `❌`). El intérprete de Python en Windows (bajo codificación local cp1252) no lograba decodificar estos caracteres, interrumpiendo el proceso del servidor.
* **Solución:** Se editaron las líneas de `print()` del módulo de Weber para utilizar textos ASCII seguros (`[OK]`, `[ERROR]`), logrando estabilidad total en el servidor.

---

## 4. Beneficios Obtenidos


1. **Despliegue Simple (Un solo clic/comando):**  
   Ahora, tanto para el desarrollo como para el uso final de los vendedores en sucursal, solo es necesario ejecutar un comando para levantar todo el sistema:
   ```bash
   python -m uvicorn app.main:app --port 8000 --reload
   ```
2. **Eliminación de CORS:** Al correr el frontend y la API bajo el mismo dominio y puerto (`http://localhost:8000`), no se realizan peticiones cruzadas, eliminando por completo potenciales bloqueos de seguridad del navegador.
3. **Resiliencia Local:** Si el servicio de Ollama está apagado en la máquina del usuario, el servidor FastAPI sigue sirviendo las calculadoras interactivas de PEISA y Weber de forma local sin problemas, permitiendo que el 90% del sistema siga operable.
