# Bitácora de Cambios v4.4.9 — Sucursales, Relevancia Weber RAG y Corrección CORS

Esta bitácora documenta las mejoras aplicadas en la versión **v4.4.9** de SOLDASUR: actualización de sucursales en el frontend, mejora en la precisión de recomendaciones del asesor Weber (filtro de relevancia semántica y ajuste de prompt), y corrección de un bug de CORS producido por la URL hardcodeada del backend.

---

## 1. Diagnóstico y Correcciones Realizadas

### A. Actualización de Sucursales en el Frontend

*   **Problema:** El frontend mostraba **4 locales** (2 por ciudad), pero la empresa opera actualmente con **1 local por ciudad**: Islas Malvinas 1950 en Río Grande y Héroes de Malvinas 4180 en Ushuaia.
*   **Solución:**
    *   Se actualizó la tarjeta de características principales de "4 Sucursales" a **"2 Sucursales"** con el texto descriptivo actualizado.
    *   En la sección "Nuestras Sucursales" se eliminó el bloque secundario de Río Grande (Av. San Martín 366 / Tel. 02964 422131) y el de Ushuaia (Gobernador Paz 665 / Tel. 02901 430886), dejando únicamente la sucursal activa de cada ciudad.

---

### B. Mejora de Precisión en Recomendaciones del Asesor Weber (RAG)

*   **Problema:** Al realizar consultas al LLM de Weber, el sistema recomendaba casi siempre **3 productos**, de los cuales al menos 1 era irrelevante para la consulta puntual. Esto ocurría por dos causas combinadas:
    1.  El buscador recuperaba `top_k=5` productos y los 3 primeros pasaban sin filtro de relevancia al LLM, incluyendo productos con baja similitud semántica.
    2.  El `system_prompt` de Weber incluía un ejemplo donde se listaban 3 productos ante una consulta genérica, lo que "entrenaba" al LLM a listar siempre 3.

*   **Solución — Backend ([weber_rag_llm.py](../RAG_engine/query/weber_rag_llm.py)):**
    *   Se redujo `top_k` de `5` a `3` en la búsqueda vectorial inicial, reduciendo el ruido semántico desde la recuperación.
    *   Se agregó la función `_filter_by_relevance()` con un umbral de `similarity_score ≥ 0.25`. Los productos que no superan el umbral se descartan antes de ser enviados al LLM. Si ningún producto supera el umbral, se garantiza al menos 1 resultado (el más relevante).
    *   El contexto enviado al LLM se redujo de **top 3** a **top 2 productos**, eliminando el tercero que era la fuente habitual de recomendaciones desubicadas.
    *   Los productos devueltos al frontend se limitaron a **máximo 2** (en lugar de siempre 3), evitando mostrar tarjetas de productos que el LLM no llegó a mencionar en la respuesta.

*   **Solución — Prompt ([configs/prompts.json](../configs/prompts.json) y fallback en [weber_rag_llm.py](../RAG_engine/query/weber_rag_llm.py)):**
    *   Se agregó la sección **"CUÁNTOS PRODUCTOS RECOMENDAR"** al `system_prompt` de Weber:
        *   Por defecto: **1 solo producto** (el más adecuado).
        *   Máximo 2 productos: **solo** si el usuario pide explícitamente "opciones", "alternativas" o "qué más hay".
        *   Prohibición explícita de listar 3 o más productos en consultas específicas.
    *   Se reemplazó el ejemplo problemático `"¿qué productos tiene weber?"` (que listaba 3 productos) por ejemplos específicos que muestran **1 solo producto recomendado**, eliminando el patrón de comportamiento que inducía al LLM a listar siempre 3.
    *   Se agregó un ejemplo de respuesta para cuando el usuario pide explícitamente opciones (2 productos máximo).
    *   Los mismos cambios se aplicaron al prompt de fallback hardcodeado en el código Python para mantener consistencia si el archivo de configuración no puede leerse.

### C. Corrección de Bug de CORS — URL Relativa en el Frontend

*   **Problema:** El frontend tenía hardcodeada la URL del backend de FastAPI como `http://127.0.0.1:8000/api/chat` en `chatbot.js`. Cuando un usuario accedía al sistema por `http://localhost:8000`, el navegador detectaba un **origen distinto** (`localhost` ≠ `127.0.0.1`) y enviaba automáticamente una solicitud de verificación previa `OPTIONS` (preflight de CORS). FastAPI, sin middleware CORS configurado para este caso, respondía `405 Method Not Allowed`, interrumpiendo la comunicación con el backend.
*   **Por qué afectaba a algunos y no a otros:** Los usuarios que accedían por `http://127.0.0.1:8000` no experimentaban el problema porque el origen de la página coincidía exactamente con el destino del fetch (mismo origen → sin preflight). Los usuarios que usaban `http://localhost:8000` sí lo experimentaban.
*   **Solución:** Se reemplazó la URL absoluta hardcodeada por una **URL relativa** en `chatbot.js`:
    ```diff
    - const BACKEND_URL = 'http://127.0.0.1:8000/api/chat';
    + const BACKEND_URL = '/api/chat';
    ```
    Con una URL relativa, el fetch siempre se dirige al mismo servidor que sirvió la página, eliminando completamente el problema de CORS independientemente de si se accede por `127.0.0.1`, `localhost`, o cualquier otra dirección de red.

### D. Reducción de Tamaño de Botones del Chatbot (−10%)

*   **Problema:** Los botones clicables del menú del chatbot (opciones de navegación, categorías del sistema experto, etc.) resultaban visualmente grandes en relación al contenido del widget, afectando la experiencia en el buscador de productos de Weber.
*   **Solución:** Se actualizó la regla `.option-btn` en `soldasur.css` para reducir el padding y el font-size un 10% respecto a los valores por defecto de Tailwind (`py-2 px-4`):
    ```css
    .option-btn {
        transition: all 0.2s;
        padding: 0.45rem 0.9rem !important; /* -10% respecto a py-2 px-4 (0.5rem / 1rem) */
        font-size: 0.9rem !important;       /* -10% respecto al tamaño base */
    }
    ```
    Se utiliza `!important` para sobreescribir las clases utilitarias de Tailwind que se aplican dinámicamente desde el JS (`py-2 px-4`), que de otro modo tendrían mayor precedencia al cargarse por CDN.
