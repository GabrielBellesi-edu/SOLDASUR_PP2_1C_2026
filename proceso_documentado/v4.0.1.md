# Análisis de Arquitectura: RAG Sintáctico (JS) vs RAG Semántico (Python) y Unificación de Criterios

Este documento detalla un hallazgo técnico crítico realizado durante las pruebas de la interfaz de chat en la transición de la v3 a la v4. Este análisis sirve como justificación arquitectónica para las decisiones de diseño del proyecto en el marco de la materia Práctica Profesionalizante II (PP2).

---

## 1. El Experimento Empírico (El Hallazgo)

Durante las pruebas de la calculadora y el chat libre de calefacción PEISA, se observó el siguiente comportamiento en una conversación limpia:

* **Caso 1 (Con errores ortográficos):**
  * **Pregunta del usuario:** *"toayeros vaño"*
  * **Resultado:** El chatbot no pudo identificar la intención específica y devolvió una recomendación genérica (productos por defecto del catálogo, como calderas o radiadores estándar). Faltó la recomendación del toallero de baño.
* **Caso 2 (Escritura correcta):**
  * **Pregunta del usuario:** *"toalleros baño"*
  * **Resultado:** El sistema respondió de forma excelente recomendando el toallero eléctrico **Domino S** y renderizando su tarjeta interactiva.

### ¿Por qué ocurre esto?
* En la arquitectura heredada, el script de frontend en JavaScript ([chatbot.js](../app/modules/chatbot/chatbot.js)) realiza un filtro manual basado en texto rígido (`msg.includes('baño')`, `msg.includes('toallero')`).
* Al escribir *"vaño"* o *"toayero"*, las funciones de filtrado locales fallan silenciosamente. El sistema se ve obligado a recurrir a un listado genérico de fallbacks (los primeros 5 productos del catálogo).
* Aunque el LLM (Ollama) es capaz de corregir ortográficamente *"vaño"* a *"baño"*, la lista de productos que el JavaScript le envió en el prompt del sistema no contenía toalleros. Por lo tanto, el LLM no pudo recomendar el producto adecuado del catálogo.

### Pruebas Adicionales sobre el Servidor Legacy (Uvicorn)
Durante las pruebas de diagnóstico para contrastar el comportamiento, se levantó el servidor backend original del **Legacy** mediante el comando recomendado en su manual:
```bash
python -m uvicorn app.main:app --reload
```
Al ejecutar esta configuración (Opción B) e interactuar con el chat abierto del frontend, se descubrió que **el chat no funciona**:
* Aunque la página web inicial levanta correctamente, al ingresar cualquier pregunta abierta en la interfaz de chat, la terminal del servidor devuelve el error:
  `INFO: 127.0.0.1:58616 - "POST /api/chat HTTP/1.1" 404 Not Found`
* **Explicación técnica:** Este fallo se debe a que en la versión Legacy, el backend FastAPI (`app/main.py`) **no implementa el endpoint `/api/chat`**. En su lugar, el frontend original de Legacy llamaba a Ollama directamente en el puerto `11434` sin pasar por la API de Python. Si el frontend (que en v3/v4 se modificó para depender de la API de Python) intenta consultar la API de chat en el Legacy, el backend responde con un error HTTP 404.

## 2. Desventajas del RAG en Frontend con JS sin Interpretación Semántica

El filtrado de catálogo en el cliente (JavaScript) utilizando comparaciones sintácticas estrictas presenta problemas graves para la experiencia del usuario final:

1. **Tolerancia a fallos nula (Rigidez ortográfica):**
   Cualquier error de dedo, intercambio de letras (ej. *v/b*, *c/s/z*) o abreviación común del usuario rompe la búsqueda del producto relevante. El sistema no puede emparejar *"vaño"* con *"baño"*.
2. **Incapacidad de comprender sinónimos o lenguaje natural:**
   Si el cliente escribe *"tengo el living congelado"*, un RAG sintáctico en JS no encontrará el término *"frío"* o *"calefacción"* a menos que los programadores agreguen de manera manual y cableada cada posible sinónimo del idioma español dentro de un condicional `if`.
3. **Mantenimiento insostenible:**
   Expandir el diccionario de palabras clave en JavaScript para capturar formas coloquiales de hablar de los usuarios genera un código sucio, acoplado y difícil de testear a largo plazo.

## 3. Disparidad de Criterios en v3 (Híbrido Desalineado)

En la versión v3 coexistían dos lógicas de diseño opuestas para el procesamiento y recomendación del catálogo:

* **PEISA (Frontend con JS):** Búsqueda rígida por palabras clave sintácticas en el navegador. Delegaba toda la "inteligencia" al LLM de forma improvisada, enviando productos por defecto cuando el filtro de JS fallaba.
* **Weber (Backend con Python):** Inicialmente estructurado para apoyarse en un motor de backend (`weber_rag_engine.py`) que realizaba búsquedas semánticas mediante embeddings vectoriales en una base de datos local de FAISS.

Esta disparidad generaba que la experiencia del usuario fuera impredecible: el chat de Weber era más tolerante a sinónimos y variaciones lingüísticas que el chat de PEISA en lenguaje natural.

---

## 4. Unificación de Criterios y Propuesta de Arquitectura en v4

Para resolver este desorden arquitectónico y unificar el comportamiento de la aplicación en la **v4**, se establecieron dos directrices claras basadas en las fortalezas de cada plataforma:

### 1. Las Calculadoras Guiadas (Sistemas Expertos por Reglas) ➔ 100% Frontend (JS)
Los flujos de preguntas paso a paso de PEISA ([peisa_expert.js](../app/modules/peisa/peisa_expert.js)) y de Weber ([weber_expert.js](../app/modules/weber/weber_expert.js)) son deterministas y se basan en datos estructurados de entrada (metros cuadrados, aislación térmica, zona geográfica). Ejecutarlos localmente en JavaScript elimina la dependencia de servidores activos, acelera la respuesta del cliente y no requiere de un procesamiento de lenguaje natural.

### 2. El Asistente Abierto de Chat (RAG Conversacional) ➔ Centralizado en Backend (Python)
Para consultas libres en lenguaje natural (ej. *"¿qué me recomiendan para una habitación helada?"* o *"toayeros vaño"*):
* El chat debe consultar obligatoriamente al endpoint unificado del backend de FastAPI (`/api/chat`).
* El backend de Python debe procesar la consulta utilizando embeddings vectoriales semánticos (a través de `SentenceTransformers` y `FAISS`).
* De esta manera, el backend recupera de forma inteligente los productos correctos (toalleros, calderas, adhesivos de construcción) sin importar errores de ortografía o modismos, inyectándolos con precisión en el contexto de Ollama antes de formular la respuesta final.

Esta separación garantiza un sistema tolerante a fallos ortográficos en el chat abierto, a la vez que mantiene operativas las calculadoras numéricas del negocio de manera local.
