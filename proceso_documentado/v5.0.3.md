# Cambios en Versión v5.0.3

Esta versión introduce mejoras clave para evitar fugas de contexto cruzadas entre marcas en el chatbot Soldy y el sistema experto unificado, garantizando una correcta delimitación de la información de producto activo y limpiando endpoints heredados y obsoletos.

## Resumen de Modificaciones

### 1. Sincronización Robusta del Contexto Activo (Frontend - `core.js`)
*   **Limpieza de Estado al Iniciar:** Se modificó `GenericExpertClient.start()` para resetear incondicionalmente `lastActiveProduct = null` al comenzar un cálculo nuevo de cualquier marca. Esto evita que consultas de seguimiento hereden productos de cálculos o marcas previas de forma incorrecta.
*   **Escaneo de Productos en Respuestas:** Se actualizó `GenericExpertClient.handleResponse(data)` para que el escaneo de productos recomendados (RAG/Catálogo), renderizado de tarjetas y sincronización de `lastActiveProduct` ocurra para cualquier nodo de tipo `"respuesta"` o `"response"`, independientemente de si el flujo del cuestionario ha finalizado por completo (`isFinal`). Esto permite mostrar tarjetas y asociar productos al contexto de forma inmediata tras mostrar una recomendación (ej: `mostrar_recomendaciones` en PEISA).

### 2. Prevención de Contaminación de Marca (RAG Backend)
*   **Enrutador PEISA (`app/main.py`):** En la llamada RAG para PEISA (`formatter == "peisa"`), se incorporó la validación del producto activo. Si el producto presente en `last_active_product` no pertenece al catálogo de PEISA (por ejemplo, si es un adhesivo de Weber), se descarta pasando `None` a la inferencia, previniendo que el LLM reciba referencias ajenas y alucine o mezcle información de marcas.
*   **Modulo Weber RAG (`rag_llm_weber.py`):** Se modificó la función principal `search_and_answer` para validar si `last_active_product` pertenece al catálogo Weber usando `get_weber_product_by_model`. Si no pertenece a Weber, no se pasa al prompt del LLM.
*   **Plantilla del Instalador (`install_brand.py`):** Se actualizó la plantilla autogenerada para el método `search_and_answer` de modo que cualquier nueva marca instalada a futuro incorpore automáticamente esta misma verificación y protección de aislamiento de producto activo.

### 3. Depuración de Código y Limpieza de Endpoints Heredados
*   **Eliminación de Endpoints Obsoletos:** Se eliminaron de forma segura los endpoints redundantes `@app.post("/expert/weber/start")` y `@app.post("/expert/weber/reply")` en `app/main.py`. Todo el procesamiento de la calculadora experta de Weber ahora se realiza a través de las rutas dinámicas unificadas `/api/expert/{brand}/start` y `/api/expert/{brand}/reply`.
